# PRD 文档模板

生成 PRD 时严格按此结构输出，每个模块必须完整填充。若某个模块确实不适用于当前产品，保留模块标题并标注"不适用"及原因。

---

# [产品名称] - 产品需求文档

## 1. 文档信息

| 字段 | 内容 |
|------|------|
| 文档版本 | v1.0 |
| 作者 | [通过对话获取，若未知写"待补充"] |
| 创建日期 | YYYY-MM-DD |
| 最后更新 | YYYY-MM-DD |
| 状态 | 草稿 / 评审中 / 已定稿 |

## 2. 产品概述与背景

### 2.1 产品定位
一句话说清楚这个产品是什么、为谁解决什么问题。

### 2.2 业务背景
为什么要做这个产品？解决了什么痛点？当前没有这个产品时用户是怎么解决的？

### 2.3 产品目标
列出 3-5 个可衡量的目标（如：提升某指标 X%、减少某流程耗时 Y 分钟、支撑 Z 万用户使用）。

### 2.4 核心价值主张
用户为什么选择这个产品而不是替代方案？一句话。

## 3. 目标用户画像

### 3.1 用户角色概览

| 角色 | 描述 | 核心诉求 |
|------|------|---------|
| [角色名] | [一句话描述] | [最重要的 1-2 个诉求] |

### 3.2 各角色详细画像

为每个角色展开：使用场景、典型行为特征、痛点、期望。

## 4. 用户故事与场景

### 4.1 核心用户旅程
描述 1-3 个端到端的核心用户旅程，覆盖主线流程。每条旅程包含：
- **用户目标**：用户想达成什么
- **前置条件**：旅程开始前的状态
- **操作步骤**：1→2→3→...
- **预期结果**：最终达到什么状态
- **异常分支**：每个步骤可能的异常情况

### 4.2 用户故事列表

用标准格式列出：
- 作为 <角色>，我希望 <功能>，以便 <价值>

按 P0/P1/P2 标记优先级。

## 5. 功能需求详情

### 5.1 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|---------|--------|------|
| F-01 | [功能名] | P0/P1/P2 | [一句话描述] |

### 5.2 各功能详细说明

对每个功能展开描述：

- **功能编号与名称**
- **优先级**：P0（必须有）/ P1（应该有）/ P2（锦上添花）
- **功能描述**：这个功能做什么
- **前置条件**：使用该功能的前提
- **主要流程**：正常操作步骤
- **异常处理**：异常情况下的行为
- **关联功能**：依赖或被哪些功能依赖

## 6. 业务规则与逻辑边界

### 6.1 核心业务规则
逐条列出业务规则，每条规则涵盖：
- **规则描述**：明确、无歧义
- **触发条件**：什么时候生效
- **规则逻辑**：怎样计算/判断
- **例外情况**：什么时候不适用

### 6.2 状态流转
若涉及实体状态变化，画状态流转描述：
- 有哪些状态
- 什么操作触发状态变化
- 哪些状态变化是单向的（不可逆）

### 6.3 边界条件与约束
- 数据边界（如：最大/最小数量、时间范围）
- 并发冲突处理（谁先提交谁生效 / 后提交覆盖）
- 幂等性要求（重复操作的结果）

## 7. 验收标准

### 7.1 功能验收
每个 P0/P1 功能列出明确的验收条件，格式：
- When <条件>, Given <前置>, Then <预期结果>

### 7.2 体验验收
- [ ] 核心用户旅程可在 X 分钟内完成
- [ ] 新用户首次使用无需培训
- [ ] [其他产品维度的体验标准]

## 8. 非功能需求

> 仅从产品视角描述，不涉及技术实现方案

### 8.1 性能体验
- 页面加载/操作响应感知标准（如：列表加载不出现明显空白等待）
- 系统容量预期（如：支撑 X 人同时使用）

### 8.2 安全与合规
- 数据权限边界（谁能看/编辑什么）
- 操作留痕需求（哪些操作需要记录日志）
- 合规要求（如需要符合某个法规）

### 8.3 兼容性与集成
- 需要与哪些现有系统/流程配合
- 数据迁移/导入需求

## 9. 范围边界

### 9.1 本期包含
明确列出本期要做哪些功能/模块。

### 9.2 本期不包含
明确列出本期不做什么，防止范围蔓延。对于每项说明"为什么不现在做"。

### 9.3 后续版本规划
值得做但因优先级/资源本期未纳入的内容，简要列出作为后续版本的输入。
