# PRD 审查技能

## 描述
从高级产品经理的视角审查 PRD（产品需求文档），涵盖业务可行性、完整性和可落地性三个维度。

## 审查框架

### 1. 业务层面审查
- **用户痛点**：是否真实且经过验证？
- **目标用户画像**：是否足够具体（规模、使用频率、技术素养）？
- **竞争格局**：存在哪些替代方案？用户为什么选择本产品而非现有方案？
- **差异化价值**：与竞品的差异是否清晰且具备防御性？
- **商业模式 / 可持续性**：如何获得资金或维持运营？

### 2. 完整性审查
- **功能范围**：功能列表是否完整？优先级划分（P0/P1/P2）是否合理？
- **边界情况**：空状态、错误状态、边界条件（零持仓、数据不可用、基金暂停交易、分红/拆分）
- **用户生命周期**：新手引导 → 日常使用 → 异常处理 → 注销/退出
- **技术范围**：产品形态（Web/App/小程序/CLI）？技术栈？部署方式？
- **数据源**：是否明确指定？是否可靠？获取成本是否可控？
- **迭代计划**：MVP 时间线、分阶段交付路线图
- **集成点**：账号体系、推送渠道、外部 API

### 3. 可落地性审查
- **数据可得性**：所需数据能否实际获取？（这是金融科技产品的头号杀手）
- **合规风险**：是否存在监管风险？（例如，中国自 2023 年起限制基金盘中实时估值）
- **模型校准**：阈值是基于数据还是拍脑袋定的？是否规划了回测？
- **公式歧义**：评分/计算公式能否直接从文档中实现？
- **工程挑战**：推送可靠性、交易日历维护、时机精度

### 4. PRD 常见问题检查清单
- [ ] 决策公式可无歧义地计算（没有"权重具体怎么用？"这类模糊地带）
- [ ] 已讨论因子之间的相关性（避免对相关信号重复计分）
- [ ] 模型阈值有校准依据或回测计划
- [ ] 数据源可用性已验证（尤其是受监管的金融数据）
- [ ] 包含竞品分析
- [ ] 用户画像包含可量化的维度（规模、使用频率、渠道偏好）
- [ ] 产品形态已明确（不仅仅是功能列表，还要说明用户如何访问）
- [ ] 金融产品的合规/法律风险已处理

## 审查交互流程

以交互式问答模式**逐个**审查 PRD 问题。审查结束后不要输出总结报告或评分卡。

每发现一个问题，按以下模式进行：

1. 以提问的形式呈现问题：
```
### 问题 N: [类别] 问题标题
**严重程度**: 🔴P0 / 🟡P1 / 🟢P2
**发现的问题**: ...
**为什么重要**: ...
**推荐方案**: ...
---
是否同意此推荐方案？(yes/no/修改意见)
```

2. 等待用户确认。用户可以：
   - `yes` / `y` / `同意` — 接受推荐方案，进入下一个问题
   - `no` / `n` / `跳过` — 跳过此问题，进入下一个
   - 提供修改意见 — 应用用户的反馈，然后进入下一个问题

3. 用户确认后，呈现下一个问题。

4. 所有问题审查完毕后，简单说明"PRD 审查完成，共发现 N 个问题，已处理 M 个。"

问题排序：优先呈现 P0 级别问题，然后是 P1，最后是 P2。
